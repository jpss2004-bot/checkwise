## 2. What ships next — Stage 2.7

### Stage 2.7 — Provider upload usability (P1 batch)

**Objective:** Make the upload flow handle multi-document submissions, give providers a real "Solicitar corrección" entry point, and bring the recurring-requirement catalog up to the same anatomy/common_errors quality as onboarding.

**Three work streams. Recommend shipping as three separate PRs.**

#### 2.7-a — Mount the existing `CorrectionRequestForm`

- **Component already exists**, fully built, with field selector + protected-field gate: `apps/web/components/checkwise/workspace/correction-request-form.tsx`. It is **currently not mounted on any route**.
- Backend wiring is mock-only: `apps/web/lib/mock/corrections.ts`.
- Stage 2.7-a should:
  1. Mount the form on the provider profile / context-bar area (likely `apps/web/components/checkwise/portal/provider-context-bar.tsx` or a new sub-route under `/portal/`).
  2. Replace the mock with a real `POST /api/v1/portal/workspaces/{id}/correction-requests` backend endpoint.
  3. Persist to a new `correction_requests` row or to `audit_events` with `actor=provider`.
  4. Notify admin via the existing Slack feedback channel (see `apps/api/app/services/feedback_service.py` for the pattern).
- **Locked decision from §18 of the plan:** Tier B scope. Provider can self-submit corrections only for `contact_email`, `contact_phone`, `contact_name`. RFC / razón social / contract reference stay support-only.
- **Risk:** low if you reuse the existing Slack notification path.

#### 2.7-b — Multi-document upload (1 Submission → N Documents)

- **Data model already supports it.** `Submission` has `documents: list[Document]` (1:N) in `apps/api/app/models/entities.py:222`.
- **The endpoint accepts only one file today**: `apps/api/app/api/v1/portal.py:1453` declares `file: UploadFile`. Change to `files: list[UploadFile]` and loop the storage + per-document prevalidation.
- **Frontend wizard** at `apps/web/components/checkwise/intake-wizard.tsx` accepts only one file in the dropzone — needs a multi-file affordance and a per-file progress list.
- **Locked decision:** 1 Submission → N Documents (contract + annex stay together). Recommended cap: N ≤ 5 files, total ≤ 30 MB per submission (current single-file cap is 15 MB).
- **Atomic semantics:** all-or-nothing per submission. If any file fails validation, the whole submission rolls back.
- **Ship behind a `multi_file_upload` feature flag** so it can be rolled back without redeploying.
- **Risk:** medium. The API change is a contract break for any code that does multi-step submission today; verify the lineage / replacement flow (`supersedes_submission_id`) still works with multi-document parents.

#### 2.7-c — Anatomy + common_errors on `RecurringRequirement`

- Stage 2 shipped these fields on `OnboardingRequirement` only.
- `RecurringRequirement` in `apps/api/app/core/compliance_catalog.py:64` needs the same treatment:
  - Add `anatomy: str = ""`, `where_to_obtain: str = ""`, `common_errors: tuple[str, ...] = ()`.
  - Per-institution defaults (the existing `_ONBOARDING_DEFAULT_*` dicts can be lifted to a shared helper module).
  - Author per-item content for the highest-volume recurring requirements: IMSS opinion, INFONAVIT certificate, ISR mensual, SAT acuse, STPS cuatrimestral.
- The calendar drawer at `apps/web/app/portal/calendar/page.tsx:421+` is where the anatomy disclosure surfaces for recurring obligations. Reuse the `DocumentGuidanceDisclosure` component pattern from `expediente-card.tsx`.

## 7. Environment & path nuance

**The repo lives at a path with an em-dash:**
```
/Users/josepablosamano/Desktop/Work — LegalShelf/checkwise/CheckWise/
```

**Next.js 15 has a known bug** where the em-dash breaks `.next/` manifest writes. Stage 1's commit `0adb1cb` introduced a workaround in `apps/web/next.config.ts`:

```python
def resolveDistDir(): str
  ...
  if path_has_non_ascii(process.cwd()):
    return f".cw-next-{sha1(cwd)[:10]}"
  return ".next"
```

So `npm run dev` and `npm run build` produce a `.cw-next-<hash>/` directory on this machine (gitignored). The override is inert on Vercel (ASCII path) where the standard `.next/` is used.

**Side effects to know:**
- Running `rm -rf .cw-next-*` while the dev server is up will crash it. Stop the preview server first.
- The dev server intermittently writes to `.cw-next-<hash>/cache/` during fast-refresh; if a build fails with `ENOENT pages-manifest.json`, stop the dev server and run `npm run build` cold.

**Backend Python venv:**
```
/Users/josepablosamano/Desktop/Work — LegalShelf/checkwise/CheckWise/apps/api/.venv/bin/python
```
Use `.venv/bin/python -m pytest tests/ -q` for the full suite (≈75s, 517 passing).
Use `.venv/bin/ruff check <files>` for lint.

**Frontend:**
- `npx tsc --noEmit` for typecheck.
- `npm run lint` (ESLint; `.cw-next-*/` is in the ignore list).
- `npm run build` for production build.

**Dev server preview tool:**
- `mcp__Claude_Preview__preview_start` with `cwd: "/Users/josepablosamano/Desktop/Work — LegalShelf/checkwise/CheckWise/frontend"` and `command: "npm run dev"`.
- Port often falls back to `60xxx` because port 3000 is sometimes occupied. The preview tool auto-allocates.
- Once started, the gate-guarded `/portal/*` routes redirect anonymous visitors to `/`. **You cannot visually verify the provider UI without an authenticated portal session.** Use static checks + jluna's seed data referenced in §18 of the plan.
