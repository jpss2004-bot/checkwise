# Metadata extraction fix — date / participants / description (2026-06-18)

## Symptom
In every client metadata workbook, three columns were blank for **all** documents:
**Fecha principal**, **Participantes**, **Descripción**. Confirmed on the flagship
master (`corporativo-industrial-anahuac`): 0/612 rows filled in each of the three.

## Root causes (three real defects)
1. **Conditional/optional fields were never built.** `build_pdf_metadata_dry_run_payload`
   iterated only `template["fields"]["required"]`. `description` is a *conditional*
   field on every rule that has it, so it never got a review item — blank even when
   the AI comprehension tier explicitly returned a value for it (the suggestion was
   silently dropped on merge).
2. **No deterministic derivation of content fields.** `_context_value_for_field`
   filled names/categories/tags but left `main_date`, `participants`,
   `provider_participant`, `taxpayer_name`, `society_participant`, `full_date_label`,
   `date_8_digits`, … permanently `pending`, even though the provider/client names
   were already in the upload context and detected dates were already in the
   classifier signals in hand.
3. **The only path that filled them was off.** `DOCUMENT_ANALYSIS_PROVIDER=disabled`
   and `COMPREHENSION_FIELD_SUGGESTIONS_ENABLED=False` in prod, so nothing populated
   the `ai_assisted` fields.

## Fix (backend only, no migration, no AI required)
- **`apps/api/app/core/metadata_rules.py`** — added a structured `fixed_description`
  to `DocumentMetadataRule`; set it for the two types the source PDF prescribes a
  fixed description for: CSF (`Constancia de Situación Fiscal`) and Registro Patronal
  (`Tarjeta de Identificación Patronal (TIP)`). Added `description` to those rules'
  conditional fields.
- **`apps/api/tools/test_pdf_metadata_dry_run.py`** — deterministic derivation layer:
  - Review items now cover **required + conditional + optional** (deduped), not just
    required. `blank` fields stay out (intentionally empty per the rulebook).
  - `participants` = provider (+ client for contracts); `provider_participant`,
    `society_participant`, `taxpayer_name` = provider; `client_participant` = client.
  - `description` = rulebook `fixed_description` where defined.
  - `main_date` / `issue_date` / `expedition_date` / `deed_date` / `start_date` from a
    **conservatively** selected detected date: used only when there is exactly one
    distinct detected date, or exactly one detected date inside the upload's expected
    period month. Otherwise left blank (better blank than wrong on a compliance
    product). `full_date_label` (`DD de mes de AAAA`) and `date_8_digits` (`DDMMAAAA`)
    are derived from that date.
  - All derived values are `prefilled_needs_review` — the reviewer still confirms;
    nothing is auto-approved. When AI comprehension is on, it now layers on top and
    only fills what the deterministic pass left blank.

Detected dates flow in via the existing precomputed-signals path used by all three
callers (live intake, async comprehension re-export, and the backfill).

## Verification
- `ruff` clean; `pytest` on the full metadata suite (90 tests) green, including new
  tests for conservative date selection and the end-to-end master-column chain.
- The 13 `requiere_aclaracion` upload-flow failures are the **pre-existing**
  env-dependent baseline (reproduced identically on a clean HEAD worktree).

## Retrofitting existing documents
New uploads get the fix automatically. Existing workbooks are regenerated by
re-running the backfill with `--force` (slots already have a `latest_metadata.xlsx`):

```
STORAGE_BACKEND=s3 .venv/bin/python -m scripts.backfill_metadata --force --apply
# or scope it: --client-id <id> / --vendor-id <id> / --period 2026-M05
```

The backfill reads the persisted `DocumentInspection` signals — participants and
description fill for every doc; dates fill where the original inspection captured them.

## Notes / follow-ups
- Most documents legitimately have **no** description (the rulebook says it is "texto
  fijo, listado de anexos o vacío"). Only the two fixed-text types fill deterministically;
  annex-listing descriptions come from the comprehension tier or manual review. If
  Legal Shelf wants standardized descriptions per compliance family, add them as
  `fixed_description` on those rules.
- Minor pre-existing: the admin metadata *preview* reader caps at 12 columns
  (`client_metadata._read_workbook_preview`), hiding the 13th column (Archivo PDF).
  Not related to the three fields. Left as-is.
