# CheckWise Whole-System Audit — Prioritized Findings Report

## Methodology

This report synthesizes 254 findings produced by a fan-out static audit across the FastAPI backend (`apps/api`) and Next.js frontend (`apps/web`), plus infra (`render.yaml`, alembic, CI). Each finding was adversarially re-verified before inclusion (several carry explicit empirical reproduction, e.g. `M(name='   ').name is None`, `_safe_filename('..') == '..'`, `lookup_recurring_by_code('REC-IMSS-2026-03')` returns `None`). Near-identical findings sharing one root cause are deduped here into a single entry that lists all locations. Severities are taken from the verified findings; theme clustering is in the `top_themes` field. No findings were invented beyond the verified list.

Spot-checks performed during synthesis confirmed the two headline correctness bugs in source: `reviewer.py:1058-1075` does `emit_reviewer_decision(...)` + `db.flush()` with **no `db.commit()`** (#6), and `config.py:652` gates only on `AUTH_JWT_SECRET == _AUTH_JWT_PLACEHOLDER` with no length floor (#7/#8).

Risk-per-effort ordering and shippable batches are in the `prioritized_batches` field. Headline numbers: **no anonymous tenant-data breach or unauthenticated RCE found**; the dominant actionable risk is notification-cutover correctness + a duplicated wrong deadline estimator, followed by latent security-hardening gaps and uncapped scans.

## CRITICAL / HIGH

### Notification cutover correctness
- **Unified notification envelope after reviewer decision is flushed but never committed** — `apps/api/app/api/v1/reviewer.py:1058` — `emit_reviewer_decision(...mode='active')` then `db.flush()` with no `db.commit()`; `get_db` does not commit on teardown, so `db.close()` rolls back the in-app/dispatch/idempotency rows for **every** human reviewer decision, and the rolled-back claim means retries aren't deduped. Auto-approval (`auto_approval.py:263-271`) correctly does emit→flush→**commit**. — Fix: add `db.commit()` after the flush inside the try; add an endpoint test asserting a `NotificationDispatch` row persists.
- **Cutover emit writes duplicate in-app notification rows alongside the legacy path** — `apps/api/app/services/notifications/fanout.py:196` — legacy `notify_*` (`submission_workflow.py:424,437`) and fabric `_write_inapp_row` both INSERT provider+client rows for the same decision, with diverging severities (legacy `approved`→green vs fabric `important`→yellow). — Fix: gate `_write_inapp_row` behind a cutover flag (or remove legacy `notify_*`); reconcile severities.
- **Synchronous email+SMS sends on request thread hold a DB connection for up to ~27s per recipient** — `apps/api/app/services/notifications/fanout.py:122` — `dispatch(mode='active')` runs SMTP (15s) then Twilio (12s) sequentially on sync handlers (`reviewer.py:1063`, `auth.py:891`, `me.py:257/455`, `admin.py:1266`, `client.py:2092`, `client_users.py:497`, `auto_approval.py:268`) before the request-scoped session commits, so a provider slowdown can exhaust the threadpool + connection pool. — Fix: defer fan-out via `background_tasks.add_task` after the response (or commit/release the connection before the calls); run email and SMS concurrently.
- **Active-mode emit sends email/SMS before the idempotency row is committed (at-least-once → duplicate sends)** — `apps/api/scripts/run_renewal_dispatch.py:125` (also `auto_approval.py:263-271`, `reviewer.py:1058-1070`) — the `notification_dispatch` claim row commits only at `:145`, after the real SMS/email went out; a crash/rollback re-sends on the next run. — Fix: durably persist the per-recipient claim before the channel send, or use a transactional outbox drained by a committed worker.

### Wrong/duplicated deadline computation (truth-in-data)
- **due_in_days_for_period derives the wrong deadline month for bimestral/cuatrimestral/annual obligations** — `apps/api/app/services/dashboard_compute.py:362` — reconstructs the deadline from the period-key month instead of the catalog's `due_month`; the SAT annual (`period_key '<year-1>-A'`, `due_month=4`) computes `date(year-1,12,17)` so it reads **~200+ days overdue all year**; bimestral/cuatrimestral surface ~1 month early. Drives the provider semaphore, attention items, upcoming deadlines, suggested actions. — Fix: thread real `due_month/due_day` (match `_calendar_deadline_iso`) and delete the period-key reconstruction.
- **Dashboard estimator and the calendar/risk engine compute different deadlines for the same obligation** — `apps/api/app/services/dashboard_compute.py:362` (+ twin `portal._due_in_days_for_period`) — `_vendor_compliance` counts via the buggy estimator while `aggregate_client_calendar` uses the true `due_month`, so a provider's rollup counts disagree with its own obligation list and the client/admin calendar. — Fix: collapse onto `calendar_risk.calendar_item_risk` + `_calendar_deadline_iso`; delete the duplicate.

### Uncapped O(clients)/O(vendors) fan-out & N+1
- **GET /admin/rollup runs an uncapped, uncached O(clients) compliance fan-out on every load** — `apps/api/app/api/v1/admin.py:632` (dedupe of #2/#3/#33; per-client `COUNT(Vendor)` N+1 at `:637` = #34) — selects every client and per-client issues ~5-6 queries; the sibling `/admin/calendar/grid` got `_GRID_CLIENT_SCAN_CAP=200` + stale-while-revalidate snapshot, `/rollup` (the ops home page) got neither. — Fix: snapshot/cap pattern; replace per-client COUNT with one `group_by(Vendor.client_id)`.
- **N+1 storm in fetch_vendor_risk_matrix / fetch_compliance_by_institution** — `apps/api/app/services/reports/blocks/data_fetchers.py:266` and `:702` — per-vendor `resolve_workspace_for_vendor` + two `_workspace_submissions` queries (+ full `Institution` scan under V2); `vendor_risk_matrix` caps `max_rows` only *after* computing all rows, `compliance_by_institution` has no cap. `build_client_context` already provides the one-query batch path. — Fix: one scoped `select(Submission)` bucketed by `vendor_id`, build `institutions_by_id` once, one workspaces IN-query, thread `prefetched_submissions`.
- **compute_renewal_actions N+1: one submissions query per renewal requirement** — `apps/api/app/services/dashboard_compute.py:777` — the lone helper in the family not using prefetch; admin renewals lane (`admin.py:3780`) ×300 vendors ≈ ~1500 queries per `GET /admin/calendar/renewals`. — Fix: add `prefetched_submissions` param; batch one submissions query per vendor.
- **Admin renewals/radar lanes fan out per vendor with no batch prefetch** — `admin.py:3779` (renewals: `resolve_workspace_for_vendor` + `compute_renewal_actions` ×300) and `admin.py:3025` (radar: `build_upcoming_deadlines_for_vendor` ×300). — Fix: bulk-resolve workspaces in one IN-query and share a portfolio submissions prefetch.
- **Admin global search runs unindexed full-scan ILIKE over all submissions with min_length=1** — `apps/api/app/services/search_service.py:197` — leading-wildcard `f_unaccent(...) ILIKE` over a 7-table join; pg_trgm/functional indexes deferred (`0051`/`0052`), `repse_folio` unindexed; admin `/search` passes no scope and `min_length=1` → full seq scan per keystroke. — Fix: add the deferred pg_trgm GIN indexes + `repse_folio` index; raise `min_length`; narrow before hydrating.

### Tenant-isolation / data-scoping
- **Client-name slug collision merges two tenants' metadata masters (cross-tenant leak)** — `apps/api/app/services/metadata_export.py:710` — export tree keyed only on `_slug(client.name)`; `Client.name` has no uniqueness (`entities.py:52`), so 'Café S.A.' vs 'Cafe SA' share `metadata_exports/<slug>/...`; `rebuild_client_master_metadata_export` aggregates both clients into one master served by `/client/metadata/download`. — Fix: key the tree on immutable unique `client.id`; backfill existing trees.
- **Duplicate (sha256) detection is global, not tenant-scoped** — `apps/api/app/services/submission_service.py:674` (+ batch `:1485`) — `WHERE Document.sha256 == ...` with no client filter; two tenants uploading the same blank template flag each other, leaking file existence across tenants and creating false review work. — Fix: join `Document→Submission` and filter `Submission.client_id`.
- **Per-vendor metadata download matches by vendor NAME, mixing same-named vendors** — `apps/api/app/services/client_metadata.py:200` — uniqueness is `(client_id, rfc)` not name; `filter_master_by_vendor(vendor.name)` merges both same-named vendors' rows. — Fix: carry RFC/vendor_id into the master and filter on it.
- **FE /client/* pages ignore ?client_id, serving the wrong tenant under internal-admin inspection** — `apps/web/app/client/notifications/page.tsx:134`, `activity/page.tsx:46`, `metadata/page.tsx:29` — none read `useUrlClientId`, so inspecting tenant B serves the default-resolved client. — Fix: thread `client_id` into the list calls + effect deps (and into mark-read mutations).

### Metadata export pipeline
- **Master rebuild runs on every single upload (full client-tree rglob + parse, O(slots) per upload)** — `apps/api/app/services/metadata_export.py:222` — `_write_metadata_workbooks` defaults `rebuild_master=True`; the intake path re-downloads every slot workbook from S3, rglob+parses every slot, rewrites+remirrors the master — quadratic over a cycle, inside a BackgroundTask. — Fix: make the intake rebuild incremental (replace only the changed slot's rows) or debounce per-client.
- **Multi-document batch submissions never generate the metadata XLSX / master row** — `apps/api/app/services/submission_service.py:1432` — `finalize_multi_document_submission` lacks the `export_metadata_table_after_upload` + event + `notify_metadata_ready` the single-file path has, so batch uploads are invisible to `/client/metadata`. — Fix: call the export per document in the batch loop.

### Privilege & boot-secret hardening
- **Pure platform_admin can take over / disable / delete an internal_admin via user-lifecycle endpoints** — `apps/api/app/api/v1/admin.py:1535` (also `:1591`, `:2326`, `:2086`) — `_assert_can_grant_role` fences *granting* privileged roles, but `update_user_status`/`reset_user_password`/`delete_user`/`revoke_user_membership` never check the **target's** roles; `reset_user_password` even returns the plaintext temp password. Latent until the first IT-only platform_admin is provisioned (which the code is built to support). — Fix: add a symmetric target-privilege guard reusing the `_PRIVILEGED_ROLES` pattern.
- **Boot guard only catches the exact placeholder — empty/weak AUTH_JWT_SECRET boots in production** — `apps/api/app/core/config.py:637/652` (dedupe of #7/#8) — `_validate_boot_security` rejects only `== _AUTH_JWT_PLACEHOLDER`; `'x'` or `''` boots and signs all HS256 JWTs, enabling offline brute force / token forgery. — Fix: also raise `InsecureBootError` when `len(AUTH_JWT_SECRET) < 32` on non-local deploys.

### Spoofable X-Forwarded-For defeats all rate limits
- **Client-controlled X-Forwarded-For lets anyone bypass the public/auth rate limits** — `apps/api/app/api/v1/contact.py:50` (+ `feedback.py:221`, `auth.py:373`, `core/rate_limit.py:291`) — `_client_ip` trusts the leftmost XFF entry and uvicorn runs with no `--forwarded-allow-ips`/ProxyHeaders, so rotating XFF gives a fresh bucket per request, defeating contact/feedback (5/hr + Slack), login, forgot-password, share-unlock; forgot-password has **no** per-account lockout (mail-bomb vector); an unresolvable IP returns `True` (never throttled). — Fix: run uvicorn with `--forwarded-allow-ips` (or ProxyHeadersMiddleware) and read `request.client.host`, or parse XFF right-to-left; add a per-email cap to forgot-password and a global per-endpoint ceiling.

### Status/escalation logic
- **Empty-password-encrypted PDFs that decrypt fine are wrongly blocked as 'encrypted'** — `apps/api/app/services/pdf_validation.py:63` — after `reader.decrypt('')` succeeds the result still sets `is_encrypted=True`, so `status_from_inspection` forces `REQUIERE_ACLARACION` and prevalidation emits `pdf_encrypted` for many legitimate SAT/IMSS/CSF PDFs. — Fix: track decrypt success; return `is_encrypted=False` on the empty-password-success branch.
- **Reporting reminders suppressed by any submission regardless of status** — `apps/api/app/services/notifications/emitters/reporting.py:146` — `_index_submissions` indexes draft/rejected/mismatch rows with no status filter, so a rejected upload silences all future t-7/t-1/t-0/overdue reminders for that period. — Fix: restrict the suppression set to satisfying statuses.

### ZIP/storage resource leaks
- **Audit/expediente ZIP is fully buffered in RAM (up to 500MB), not streamed** — `apps/api/app/services/audit_package.py:334/335` (+ `expediente_zip.py:203`) — builds the whole `io.BytesIO` archive (each file via `src.read()`) then a single `yield buffer.read()`; concurrent large downloads can OOM-kill a single worker. — Fix: zipstream-ng / SpooledTemporaryFile chunked yields; stream S3 bodies directly.
- **S3 open_for_read leaks one temp file per ZIP entry (disk exhaustion)** — `apps/api/app/services/storage.py:339` (+ ZIP loops `audit_package.py:343`, `expediente_zip.py:210`, proxy `portal.py:2370`, `client.py:2417`) — `tempfile.mkstemp` per object, never deleted; a 500MB download leaves ~500MB of orphans until a deploy wipes `/tmp`; when `/tmp` fills, uploads+downloads fail with ENOSPC. — Fix: try/finally unlink (only when the backend materialized a temp copy), or stream the S3 body without a temp file.

## MEDIUM

### Rate limits & abuse on heavy endpoints
- **client_add_provider has no rate limit (email/SMS spam + row flood)** — `client.py:1973` — fires welcome email + invitation SMS to caller-controlled contact on every call; the only heavyweight side-effecting client endpoint with no limiter. — Fix: `enforce_export_rate_limit` (or a dedicated invite bucket) before any write/send.
- **Unbounded provider uploads trigger unrate-limited AI cost** — `portal.py:2808` — submission + batch endpoints schedule the full intake pipeline incl. inline shadow AI with no per-user throttle (Wise `/ask` at `:4503` does throttle). — Fix: per-owner upload throttle returning 429.
- **Portal Wise rate limit silently bypassed when workspace.owner_user_id is null** — `portal.py:4503` — limit is inside `if workspace.owner_user_id:` but the column is nullable and cookie auth doesn't require an owner. — Fix: always throttle with `workspace.owner_user_id or f"workspace:{workspace.id}"`.
- **Report export / admin expediente / admin+client audit-package preview/tree / phone-verify confirm lack rate limits** — `reports.py:1612`, `admin.py:5667/5694`, `client.py:3376`, `me.py:397` — heavy background renders, ZIP streams, and full-portfolio document scans run unthrottled though sibling paths and `enforce_export_rate_limit` exist. — Fix: wire the existing limiters onto each.

### Security (medium)
- **update_workspace_admin sets owner_user_id to an arbitrary unvalidated user** — `admin.py:2679` — writes `payload.owner_user_id` with no existence/active/deleted check; `owner_user_id` is the portal auth carrier, so a typo/over-broad operator grants cross-tenant document access. — Fix: validate the user exists/active/non-deleted; reject internal/client admins.
- **Password reset/change does not invalidate existing JWT sessions** — `auth.py:946` (+ `:1077`) — no `password_changed_at`/`token_version` on `User`; a stolen JWT survives the canonical 'account compromised' reset until natural expiry (24h). — Fix: add `tokens_valid_after`, reject tokens with `iat` before it.
- **Share-unlock cookie never gets Secure in production** — `shares.py:362/368` — uses `getattr(settings, "COOKIE_SECURE", False)`; the real attr is `settings.cookie_secure`, so the HMAC unlock cookie is always minted without Secure. — Fix: use `settings.cookie_secure`.
- **WhatsApp transactional sends target unverified user.phone** — `transactional_whatsapp.py:104/198/247` — reads legacy `User.phone` (never OTP-verified) instead of `phone_e164`; both a security gap (unverified destination) and a bug (OTP-verified users get no WhatsApp). — Fix: read `phone_e164` and gate on `phone_verified_at`.
- **WhatsApp reviewer-decision label dict keyed on wrong action values** — `whatsapp_templates.py:50` — `_DECISION_LABEL` keys (`approved`/`rejected`/…) never match the `ReviewerAction.value` (`approve`/`reject`/…) passed in, so providers see raw English `approve`/`reject`. — Fix: rekey to canonical values; add a unit test.
- **Plaintext OTP code written to logs in WhatsApp dry-run mode** — `whatsapp_delivery.py:176` — `WHATSAPP_DRY_RUN` logs full `components` incl. the 6-digit code. — Fix: redact; reject `WHATSAPP_DRY_RUN` in production.
- **In-memory rate limiter silently weakens all caps past one worker** — `core/rate_limit.py:211` (+ `spend_limiter.py:36`) — falls back to a per-process dict when `REDIS_URL` is unset; scaling to N workers multiplies every cap (login, forgot-password, share-unlock, AI cost) by N with only a log warning. — Fix: fail-closed at boot when non-local + workers>1 + Redis unset.
- **Prompt injection: PDF text and Wise history inlined without delimiting** — `document_analysis/expediente.py:383` (PDF-derived text JSON-dumped into the instruction turn) and `wise/ai.py:406` (client-asserted `assistant` turns, ~24KB bypassing the 500-char prompt cap). — Fix: wrap untrusted data in an explicit delimiter + system clause; replay only server-recorded turns; cap total history chars.
- **Detected wrong-RFC document routed more leniently than no-RFC** — `submission_service.py:270` — `RFC_ALIGNMENT_MISMATCH` (strong 'wrong company' signal) lands in `PENDIENTE_REVISION` while `RFC_ALIGNMENT_ABSENT` escalates to `REQUIERE_ACLARACION`. — Fix: add the mismatch variants to the escalation condition.

### Concurrency / races
- **TOCTOU: concurrent reviewer decision and auto-approval can both terminalize one submission** — `submission_workflow.py:148` — check-then-act with no row lock / version column; both pass `_validate_transition` and apply conflicting terminal transitions, corrupting the audit trail. — Fix: `with_for_update()` reload (or `version_id_col`).
- **version_number MAX+1 races the unique constraint** — `reports/executor.py:337` (+ `report_service.py:631`) — concurrent generate/refresh both compute the same `next_n`; second commit raises uncaught `IntegrityError` (500, lost AI work). — Fix: `FOR UPDATE` on the parent or bounded retry.
- **Concurrent master/latest rebuilds race on fixed paths** — `metadata_export.py:411` — non-atomic zip writes to fixed `client_master_metadata.xlsx` / `latest_metadata.xlsx`; readers can hit `BadZipFile` (422), writers lose updates. — Fix: temp-write + `os.replace`; per-client advisory lock.
- **Double-submit creates parallel non-genesis leaves for one slot** — `portal.py:2960` — the `ux_submissions_active_slot` index only covers genesis rows; two async receipts both non-genesis escape it. — Fix: serialize per-slot writes (`FOR UPDATE` / advisory lock).
- **Concurrent first-time period/requirement create 500s on the unique constraint** — `requirement_service.py:84` — SELECT-miss-then-INSERT with no `IntegrityError` handling on `Period`/`Requirement`/`RequirementVersion`. — Fix: SAVEPOINT + re-SELECT (or `ON CONFLICT DO NOTHING`).

### Bugs (medium)

- **Multi-client client_admin gets spurious 404 marking a notification read without ?client_id** — `client.py:3060` — `_resolve_client_id` returns `visible[0]`, so 2nd..Nth client notifications 404. — Fix: verify `row.client_id in _visible_client_ids_for_user(...)`.

- **Calendar error-retry sets year to its current value, so React bails and never refetches** — `apps/web/app/client/calendar/page.tsx:327` — `setYear(y => y)` is a no-op. — Fix: a `reloadKey` state in the fetch deps.
- **Empty load_type/institution_code on fresh mount cause backend 422 with no inline guidance** — `intake-wizard.tsx:168` — native `<select value="">` shows the first option but state stays `""`; step-0 validation never checks them. — Fix: step-0 validation + a disabled placeholder option.

### Performance / resilience (medium)
- **build_entries re-runs the full query set inside the streaming generator** — `audit_package.py:318` — caller already computed entries for the cap/manifest, then `stream_audit_package` re-derives them (workspaces + full Institution/Requirement scans + submissions + documents). — Fix: pass the resolved entries list in.
- **ZIP arcname can contain a '..' path segment (Zip-Slip on extraction)** — `audit_package.py:610` — `_safe_filename` keeps dots, so `..` survives into the arcname; naive extractors escape the target. — Fix: reject/rewrite `.`/`..` segments in both `audit_package.py` and `expediente_zip.py`; sanitize `period_key`.
- **Concurrent uploads of same filename race on a shared temp path** — `storage.py:134` — `_stream_to_temp` derives the temp path from `sha1(safe_name)` only; two concurrent `documento.pdf` uploads interleave writes → corrupt bytes + wrong sha256. — Fix: `tempfile.mkstemp(dir=...)` for a unique staging path.
- **Renewal email/WhatsApp dropped permanently on transient outage** — `transactional_whatsapp.py:118` — dedup `RenewalReminder` committed alongside a non-raising `status='failed'` send; next run skips on dedup, no retry. — Fix: per-channel status + bounded reconcile cron.
- **PDF export has no timeout/concurrency bound; Chromium install can stampede** — `reports/export.py:248` — N concurrent exports spawn N ~200MB Chromium processes (512MB host) and each missing-binary failure races `_install_chromium`. — Fix: process-wide semaphore, install lock, explicit timeouts, rate limit.
- **Several FE fetches lack timeouts (hang on stalled R2/backend)** — `portal.ts:665`, `reviewer.ts:26`, `search.ts:48`, `reports.ts:689` — bare `fetch()` with no AbortController, unlike the timeout-bounded `fetchJson`. — Fix: add an AbortController ceiling throwing the typed `*ApiError`.
- **Admin calendar snapshot has no cron refresh; first read after a date boundary serves stale risk** — `calendar_snapshot.py:41` — today-relative risk baked into the cache, refresh is read-triggered only, no per-client error isolation (`:112`, one bad client poisons the year + silent background failure). — Fix: treat `computed_at.date() < today` as hard-stale; per-client try/except + advisory lock.

### Infra / migrations (medium)
- **Cron services build without the dependency lock** — `render.yaml:319/405/477` — bare `pip install -e .` re-opens the unpinned drift that crash-looped prod; a broken intake-reconcile cron strands `recibido` receipts. — Fix: append `-c requirements.lock`.
- **CI installs unpinned deps while prod deploys the lock** — `.github/workflows/ci.yml:26` (+ `security.yml:68`) — CI green doesn't validate the shipped set. — Fix: install with `-c requirements.lock`; fail CI on a stale lock.
- **Alembic runs the whole chain in one transaction despite autocommit-block migrations** — `alembic/env.py:64` — no `transaction_per_migration=True`; entering an autocommit_block commits all preceding pending migrations, so a later failure leaves schema/version inconsistent. — Fix: set `transaction_per_migration=True`.
- **Migration 0044 downgrade now destroys operator-granted platform_admin memberships** — `0044_backfill_platform_admin.py:63` — blanket `DELETE WHERE role='platform_admin'` is no longer safe now that the admin grant endpoint mints them. — Fix: make the downgrade non-destructive.
- **Empty CORS_ORIGINS silently disables all cross-origin auth with no boot warning** — `config.py:473/34` (+ `:540`) — `sync:false` + empty value → `allow_origins=[]` blocks every credentialed request and empties CSRF origins, with no soft warning (unlike FRONTEND_BASE_URL/REDIS_URL). — Fix: add a boot soft warning when `cors_origins_list` is empty on non-local.
- **Crons run independent of migrations** — `render.yaml:312/254` — only the web service has `preDeployCommand: alembic upgrade head`; a cron tick can run new code against an un-migrated schema. — Fix: each cron asserts the expected alembic head and no-ops-with-alert on mismatch.

## LOW

### Security (low)
- **Login lockout response enables account enumeration** — `auth.py:673` — 429 lockout detail only for existing active accounts vs generic 401 elsewhere = an existence oracle. — Fix: return the generic 401 for the locked case.
- **404-vs-403 leaks cross-tenant client existence to client_admin** — `client.py:190` — `_resolve_client_id` distinguishes missing (404) from forbidden (403); siblings use a uniform 404. — Fix: uniform 404 for non-internal callers.
- **reset-password/preview GET has no rate limit (token-validation oracle)** — `auth.py:901` — unauthenticated true/false token probe with no throttle. — Fix: per-IP limit; consider throttling POST too.
- **Share-info probe discloses the title of a password-protected share** — `shares.py:324` — returns `Report.title` before the password is supplied. — Fix: suppress the real title until a valid unlock cookie.
- **Legacy POST /submissions trusts browser-posted tenant identity** — `endpoints.py:196` — caller-chosen client/vendor get_or_create, gated only by an env flag + self-or-internal-admin; risky if `EXPOSE_LEGACY_SUBMISSIONS` is ever set in prod. — Fix: make the flag local-only and non-overridable.
- **Slack mrkdwn injection / urlopen SSRF / SMTP CR-LF crash** — `feedback_service.py:413`, `contact_service.py:153`, `email_delivery.py:376` — user-supplied url/name/message interpolated into Slack mrkdwn unescaped (staff-phishing link forgery); Slack delivery uses `urlopen` on a config URL with no host allowlist; SMTP header assignment outside try/except crashes on CR/LF. — Fix: escape Slack specials + validate url; allowlist `hooks.slack.com`; move header construction inside try/except.
- **Password-reset token + email travel in a GET query string** — `apps/web/lib/api/auth.ts:132` — exposed via history/logs/Referer. — Fix: POST the token in the body or set `Referrer-Policy: no-referrer`.
- **Admin vendor ZIP / metadata-export list do blocking IO and accept free-form status** — `admin.py:5667/5076/2644` — no export rate limit, per-row `exists()` stats, and `WorkspaceUpdate/ClientUpdate/VendorCreate/VendorUpdate.status` are plain `str`. — Fix: throttle; skip local stats on the durable mirror; constrain status to a Literal.
